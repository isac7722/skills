## 개요

**clarify**는 모호하거나 불완전한 요구사항을 구조화된 질문 프로세스를 통해 명확한 스펙으로 변환하는 스킬입니다.

구현에 들어가기 전에 요구사항을 정확히 파악함으로써, 잘못된 구현이나 스코프 확장을 방지합니다.

## 트리거 조건

다음 상황에서 clarify 스킬이 활성화됩니다:

| 방법 | 예시 |
|------|------|
| 슬래시 명령어 | `/clarify` |
| 요구사항 명확화 요청 | "요구사항 정리해줘", "clarify requirements" |
| 구체화 요청 | "이거 좀 더 명확하게 해줘", "specify requirements" |
| 의도 확인 요청 | "내가 뭘 원하는 건지 정리해줘", "what do I mean" |
| 모호한 요청 인식 | 요청이 여러 해석이 가능한 경우 자동 트리거 |

## 동작 원리: 4단계 프로토콜

### Phase 1: 원본 요구사항 캡처

사용자의 원래 요청을 그대로 기록합니다.

```markdown
## Original Requirement
"로그인 기능 추가해줘"
```

그리고 다음을 식별합니다:
- 불명확하거나 미지정된 부분이 무엇인지
- 어떤 가정이 필요한지
- 어떤 결정이 해석에 맡겨져 있는지

### Phase 2: 반복적 명확화

식별된 모호한 부분에 대해 **하나씩** 질문을 던집니다. 모든 측면이 명확해질 때까지 반복합니다.

```
모호한 부분이 남아있는 동안:
    가장 중요한 모호성을 식별
    → 명확화 질문 (선택지 제공)
    → 답변 반영하여 요구사항 업데이트
    → 새로운 모호성 확인
```

질문은 선택지를 포함한 형태로 제공됩니다:

```
질문: "어떤 인증 방식을 사용할까요?"
선택지:
  - 이메일/비밀번호: 전통적인 로그인 방식
  - OAuth: Google, GitHub 등 소셜 로그인
  - 매직 링크: 이메일로 로그인 링크 전송
```

### Phase 3: Before/After 비교

모든 질문이 완료되면, 원본과 명확화된 요구사항을 비교하여 보여줍니다.

```markdown
## Requirement Clarification Summary

### Before (Original)
"로그인 기능 추가해줘"

### After (Clarified)
**Goal**: 이메일/비밀번호 기반 로그인 및 셀프 회원가입 기능 구현
**Scope**: 로그인, 로그아웃, 회원가입, 비밀번호 재설정
**Constraints**: 24시간 세션, bcrypt 암호화, 5회 시도 제한
**Success Criteria**: 사용자가 회원가입, 로그인, 로그아웃, 비밀번호 재설정 가능

**Decisions Made**:
| 질문 | 결정 |
|------|------|
| 인증 방식 | 이메일/비밀번호 |
| 회원가입 포함 여부 | 예, 셀프 가입 |
| 세션 유지 시간 | 24시간 |
| 비밀번호 요구사항 | 최소 8자, 대소문자 혼합 |
```

### Phase 4: 저장 옵션

명확화된 결과를 파일로 저장할지 묻습니다.

- 저장 위치: `requirements/` 디렉토리 (또는 프로젝트에 적합한 위치)
- 파일명: 주제 기반 (예: `auth-feature-requirements.md`)
- 형식: Before/After 구조의 마크다운

## 질문 설계 원칙

clarify가 던지는 질문은 다음 원칙을 따릅니다:

| 원칙 | 설명 | 예시 |
|------|------|------|
| **구체적 > 일반적** | 추상적 선호가 아닌 구체적 디테일을 물음 | "DB는 뭘 쓸까요?" (O) vs "기술 선호가 어떻게 되나요?" (X) |
| **선택지 > 열린 질문** | 2~4개 선택지 제공 (인식 > 회상) | "A, B, C 중 어떤 것?" (O) vs "어떻게 하고 싶으세요?" (X) |
| **한 번에 하나** | 여러 질문을 묶지 않음 | 인증 방식만 먼저, 세션 시간은 다음에 |
| **중립적 프레이밍** | 특정 선택지에 편향 없이 제시 | 모든 옵션을 동등하게 설명 |

## 모호성 카테고리

clarify가 탐색하는 주요 모호성 유형:

| 카테고리 | 탐색하는 질문들 |
|----------|----------------|
| **범위 (Scope)** | 뭐가 포함되나요? 뭐가 명시적으로 제외되나요? |
| **동작 (Behavior)** | 엣지 케이스는? 에러 시나리오는? |
| **인터페이스 (Interface)** | 누가/무엇이 상호작용하나요? 어떻게? |
| **데이터 (Data)** | 입력값은? 출력값은? 형식은? |
| **제약 (Constraints)** | 성능 요구사항? 호환성? |
| **우선순위 (Priority)** | 필수 vs 있으면 좋은 것? |

## 실전 사용 예시

### 예시 1: 모호한 기능 요청

**원본**: "로그인 기능 추가해줘"

**명확화 과정**:

| 순서 | 질문 | 답변 |
|------|------|------|
| 1 | 인증 방식은? | 이메일/비밀번호 |
| 2 | 회원가입도 포함? | 예, 셀프 가입 |
| 3 | 세션 유지 시간은? | 24시간 |
| 4 | 비밀번호 요구사항은? | 최소 8자, 대소문자 혼합 |

**명확화된 결과**:
- **Goal**: 이메일/비밀번호 기반 로그인 + 셀프 회원가입
- **Scope**: 로그인, 로그아웃, 회원가입, 비밀번호 재설정
- **Constraints**: 24시간 세션, bcrypt 암호화, 5회 시도 제한
- **Success**: 사용자가 회원가입/로그인/로그아웃/비밀번호 재설정 가능

### 예시 2: 모호한 버그 리포트

**원본**: "내보내기가 안 돼요"

**명확화 과정**:

| 순서 | 질문 | 답변 |
|------|------|------|
| 1 | 어떤 내보내기? | CSV |
| 2 | 어떤 현상? | 빈 파일이 생성됨 |
| 3 | 언제부터? | v2.1 업데이트 이후 |
| 4 | 재현 방법? | 아무 리포트나 내보내기 |

**명확화된 결과**:
- **Goal**: CSV 내보내기 시 빈 파일 생성되는 버그 수정
- **Scope**: CSV만 해당, 다른 형식은 정상
- **Constraints**: v2.1 이후 발생한 리그레션
- **Success**: CSV에 UI와 일치하는 올바른 데이터가 포함됨

### 예시 3: 모호한 리팩토링 요청

**원본**: "이 코드 좀 개선해줘"

**명확화 과정**:

| 순서 | 질문 | 답변 |
|------|------|------|
| 1 | 어떤 측면을 개선? | 가독성 |
| 2 | 대상 범위는? | 이 파일만 |
| 3 | API 변경 허용? | 아니요, 내부만 |
| 4 | 테스트도 수정? | 필요하면 |

**명확화된 결과**:
- **Goal**: 해당 파일의 가독성 개선 리팩토링
- **Scope**: 단일 파일, 내부 구현만
- **Constraints**: 외부 API 변경 불가
- **Success**: 동일한 동작, 개선된 가독성

## 출력 형식

명확화 완료 시 아래 구조로 결과가 제공됩니다:

```markdown
## Requirement Clarification Summary

### Before (Original)
"{원본 요청 그대로}"

### After (Clarified)
**Goal**: [정확한 목표 설명]
**Scope**: [포함/제외 범위]
**Constraints**: [제한사항, 요구사항, 선호사항]
**Success Criteria**: [완료 기준]

**Decisions Made**:
| 질문 | 결정 |
|------|------|
| [모호한 부분 1] | [선택된 옵션] |
| [모호한 부분 2] | [선택된 옵션] |
```

## 저장 옵션

명확화 결과는 파일로 저장할 수 있습니다:

- **저장 위치**: `requirements/` 디렉토리 (기본값) 또는 프로젝트에 맞는 위치
- **파일명**: 주제 기반으로 자동 생성 (예: `auth-feature-requirements.md`)
- **형식**: Before/After 구조의 마크다운 파일
- 저장 여부는 Phase 4에서 선택지로 제공됩니다

## 핵심 규칙

| 규칙 | 설명 |
|------|------|
| **가정하지 않기** | 불명확한 부분은 반드시 질문으로 확인 |
| **의도 보존** | 사용자의 원래 의도를 정제하되, 방향을 바꾸지 않음 |
| **최소한의 질문** | 꼭 필요한 것만 물어봄 |
| **답변 존중** | 사용자의 결정을 그대로 수용 |
| **변경 추적** | 항상 Before/After 비교를 제공 |

## 팁과 활용법

- **구현 전에 사용하세요**: clarify는 코드를 작성하기 전에 요구사항을 정리하는 도구입니다. 구현 중간에 사용하면 이미 작성한 코드와 충돌할 수 있습니다.

- **팀 커뮤니케이션에 활용하세요**: 명확화된 결과를 파일로 저장하면, 팀원들과 요구사항을 공유하는 문서로 활용할 수 있습니다.

- **복잡한 기능일수록 효과적입니다**: 단순한 작업보다는 여러 해석이 가능한 복잡한 기능 요청에서 가장 큰 효과를 발휘합니다.

- **자연어로 자유롭게 사용하세요**: `/clarify` 명령어 외에도 "정리해줘", "명확하게 해줘" 같은 자연어 표현으로도 트리거됩니다.
