# PR Convention

이 프로젝트의 Pull Request 작성 규칙입니다.

---

## Goal

리뷰어가 PR을 열었을 때 **30초 안에** 이 PR이 왜 필요하고, 무엇이 바뀌고, 어떻게 검증하는지 파악할 수 있어야 합니다.

이를 통해:
- 리뷰 속도와 품질 향상
- 맥락 전달 비용 최소화
- 일관된 PR 히스토리 유지

---

## PR 제목

### 형식

```
[티켓번호] <type>(도메인): <한국어 설명>
```

- **티켓번호**: git 브랜치명에서 자동 추출 (예: `feat/AHD-123-social-login` → `AHD-123`)
- **type**: Conventional Commits 타입
- **도메인**: 브랜치명에서 추론하거나 변경 파일의 주요 모듈명
- **설명**: 한국어로 작성

### 추출 규칙

- **티켓번호**: git 브랜치명에서 `AHD-숫자` 패턴을 추출
- **type**: 브랜치명이 아닌 **실제 변경 내용(diff, 커밋 이력)**을 분석하여 결정

```
브랜치: feat/AHD-123-social-login
  → 티켓: AHD-123 (브랜치에서 추출)
  → type: feat (diff 분석: 새 기능 추가)
  → 제목: [AHD-123] feat(auth): 소셜 로그인 추가

브랜치: feat/AHD-456-payment-duplicate
  → 티켓: AHD-456 (브랜치에서 추출)
  → type: fix (diff 분석: 버그 수정이 주 내용)
  → 제목: [AHD-456] fix(payment): 결제 중복 요청 방지

브랜치: refactor/improve-auth-middleware
  → 티켓: 없음
  → type: refactor (diff 분석: 동작 변화 없는 구조 개선)
  → 제목: refactor(auth): 인증 미들웨어 구조 분리
```

브랜치 prefix(`feat/`, `fix/` 등)는 참고만 하고, **type은 항상 실제 변경 내용을 기준**으로 판단합니다.
티켓번호가 없는 브랜치는 `[티켓번호]` 부분을 생략합니다.

### Rules

- **70자 이내**로 작성
- **동사로 시작**: "추가", "수정", "개선", "제거" 등
- **마침표 없음**
- **한국어로 작성**
- **티켓번호는 브랜치에서 추출** — 임의로 만들지 않음

### Type 선택 기준

| Type | 언제 사용 |
|------|----------|
| `feat` | 사용자가 새로운 걸 할 수 있게 됨 |
| `fix` | 잘못된 동작 / 크래시 수정 |
| `refactor` | 동작 변화 없이 코드 구조 개선 |
| `docs` | 문서만 변경 |
| `style` | 코드 포맷팅, 세미콜론 등 (로직 변화 없음) |
| `test` | 테스트 추가/수정 |
| `chore` | 유지보수 작업 |
| `perf` | 성능 개선 |
| `build` | 빌드 시스템, 의존성 변경 |
| `ci` | CI/CD 설정 변경 |

### 좋은 예시

```
[AHD-123] feat(auth): 카카오 소셜 로그인 추가
[AHD-456] fix(payment): 결제 중복 요청 방지 처리
refactor(auth): 인증 미들웨어 구조 분리
[AHD-789] docs(api): API 문서 엔드포인트 설명 보강
```

### 나쁜 예시

```
update                               # 타입 없음, 모호
feat: add feature.                   # 영어, 마침표, 도메인 없음
[AHD-999] fix: 버그 수정             # 어떤 버그인지 불명확
feat: 로그인 관련 여러 가지 수정       # 여러 목적 혼합, 도메인 없음
[AHD-123] feat(auth): 소셜 로그인.   # 마침표
```

## PR 본문 구조

아래 3개 섹션을 **반드시** 포함합니다:

```markdown
## Summary
<!-- 이 PR이 왜 필요하고 무엇을 바꾸는지 1-3줄 -->

## Changes
<!-- 논리적 그룹별로 변경사항 정리 -->

## Notes for Reviewer
<!-- 중점적으로 봐야 할 부분, 의도적 판단, 관련 이슈 -->
```

---

## 각 섹션 작성 가이드

### Summary

- **1-3개 bullet point**로 핵심만 전달
- "왜" → "무엇을" 순서로 작성
- 관련 이슈가 있으면 `Closes AHD-123` 포함

**좋은 예시:**
```markdown
## Summary
- 결제 과정에서 네트워크 타임아웃 시 중복 결제가 발생하는 버그 수정
- 멱등키 기반 중복 요청 방지 로직 추가

Closes AHD-456
```

**나쁜 예시:**
```markdown
## Summary
- 결제 관련 수정
```

### Changes

- **논리적 그룹별**로 묶어서 정리 (파일 단위 나열 지양)
- 변경의 "의도"가 드러나도록 작성

**좋은 예시:**
```markdown
## Changes

### 멱등키 기반 중복 방지
- `PaymentService.process()`에 멱등키 검증 로직 추가
- Redis에 멱등키 캐시 (TTL: 24h)

### 에러 핸들링 개선
- 타임아웃 시 자동 재시도 제거 → 사용자에게 재시도 안내
- `PaymentTimeoutError` 커스텀 예외 추가
```

**나쁜 예시:**
```markdown
## Changes
- payment_service.py 수정
- redis_client.py 수정
- exceptions.py 추가
```

### Notes for Reviewer

- **중점적으로 봐야 할 부분**: "이 파일/로직을 집중적으로 봐주세요"
- **의도적 판단**: "A 대신 B를 선택한 이유는..."
- **셀프 리뷰 발견 사항**: Warning/Info 항목 포함
- **관련 PR/이슈**: 연관된 다른 작업물 링크

## PR 크기 가이드

| 변경 줄 수 | 판정 | 권장 액션 |
|-----------|------|----------|
| ~200줄 | 이상적 | 그대로 진행 |
| 200-400줄 | 허용 | 가능하면 분할 고려 |
| 400줄 초과 | 경고 | 분할 강력 권장 |

**예외**: 문서, 자동 생성 코드, 설정 파일은 줄 수에서 제외하여 판단합니다.

### 분할이 어려운 경우

- 하나의 기능이 여러 레이어에 걸쳐 있을 때
- 리팩토링과 기능 추가가 불가분일 때

이 경우 **Changes 섹션에서 논리적 그룹을 명확히** 나누어 리뷰어가 단계별로 이해할 수 있게 합니다.

---

## 범위 원칙

### 하나의 PR = 하나의 목적

| 좋은 예 | 나쁜 예 |
|--------|--------|
| 소셜 로그인 추가 | 소셜 로그인 + 코드 포맷팅 + 린터 설정 |
| 결제 버그 수정 | 결제 버그 수정 + 성능 개선 + 테스트 추가 |

### 혼합이 감지되면

목적이 섞인 변경은 별도 PR로 분리합니다:
1. 리팩토링 먼저 PR
2. 기능 추가 PR (리팩토링 위에)

## 셀프 리뷰 체크리스트

PR 생성 전 아래 항목을 자동 검사합니다:

| 체크 | 대상 패턴 | 심각도 |
|------|----------|--------|
| 디버그 코드 | `console.log`, `print(`, `debugger`, `breakpoint()`, `import pdb` | Warning |
| 민감 정보 | `.env`, `credentials`, `secret`, `token` 파일 변경 | Error (차단) |
| TODO/FIXME | `TODO`, `FIXME`, `HACK`, `XXX` | Info |
| 테스트 누락 | feat/fix 변경인데 테스트 파일 변경 없음 | Info |

- **Error** → PR 생성 차단. 수정 필수
- **Warning** → 표시하되 진행 허용. Notes for Reviewer에 기록
- **Info** → 참고 사항. Notes for Reviewer에 기록

---

## 리뷰어 지정 가이드

- 해당 도메인/모듈에 가장 익숙한 사람을 지정
- 1-2명이 적절 (너무 많으면 책임 분산)
- 드래프트 PR은 리뷰어 지정 없이 먼저 올려서 방향 확인 가능

---

## 언어 규칙

- PR **제목**: 한국어 (Conventional Commits type은 영어)
- PR **본문**: 한국어
- **코드 내 주석/변수명**: 프로젝트 컨벤션에 따름
